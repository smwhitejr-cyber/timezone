AccountCode Handoff - MemberAccountUpdate

Workspace note created April 27, 2026. Based on traced OmniScript code paths, not naming assumptions.

Executive Summary

In the MemberAccountUpdateDetail XML flow, AccountCode is used to identify the correct Omni member account occurrence. It is not treated as a generic update field and is not persisted by this flow as its own DE###.

The value is passed to AcctCodeXLate(AccountCode, MemberID). That function first tries to use AccountCode directly as an Omni plan/account occurrence id. If that does not work, it treats the value as a product/account type, such as IND, and searches the member's occurrences to find the occurrence whose PTPH700 product matches the supplied AccountCode.

The right account is found by either AccountCode as PlanId + MemberID, or by MemberID + AccountCode matched to PTPH700, returning the actual PlanId/occurrence.

Confirmed Behavior

1. Where AccountCode is first received

{FN}VT.TXNMACCUPD reads AccountCode from the XML token Transaction.MemberAccountUpdateDetail.AccountCode.Data. 

ElseIf (XmlTagType = 'Transaction.MemberAccountUpdateDetail.AccountCode.Data');
    AccountCode=OcXml_Data();

2. AccountCode is not added to the update DE container

Other child tags under MemberAccountUpdateDetail are added to the generic data element container using AddDeItem, but AccountCode is handled separately before that generic path. This is why AccountCode is a selector/context value, not one of the update fields.

3. AccountCode translation

The script calls AcctCodeXLate and reads the resolved values from OcPram. 

AcctCodeXLate(MemberId: MemberNumber AccountCode:AccountCode);
EmployerNumber=OcPram_Value('Plan');
Product=OcPram_Value('Product');
CrossPlanFlag=OcPram_Value('CrossPlan');

AcctCodeXLate first tries the supplied value as a plan/member occurrence: 

If PtPhObj_Get(Plan:AccountCode PartID:MemberID) = true;
  ReturnPlan = AccountCode;
  ReturnProduct = PtPhObj_De(700);
  ReturnSubProduct = PtPhObj_De(701);

If that lookup fails, it treats AccountCode as a product/account type and searches occurrences: 

BaI2Obj_View(Plan:'000001' ViewType:'PH' ViewPos:'1' KeyText:MemberID);
Loop While ((BaI2Obj_Next()) AND (FoundPlan = false));
  CurrentI2 = OcText_Trim(BaI2Obj_De(035));
  If ((PtPhObj_Get(Plan: CurrentI2 PartID: MemberID)) And (PtPhObj_De(700) = AccountCode));
    ReturnPlan = CurrentI2;
    ReturnProduct = AccountCode;
    ReturnSubProduct = PtPhObj_De(701);

Data Flow

XML Transaction type="MemberAccountUpdate"
  -> {ED}TRANBUILD parses transaction type
  -> {CT}TRANBUILD maps MemberAccountUpdate to {FN}VT.TXNMACCUPD.TxnMaccUpd
  -> {FN}VT.TXNMACCUPD reads AccountCode and MemberNumber
  -> AcctCodeXLate(AccountCode, MemberID)
       Case A: AccountCode is a valid PlanId/account occurrence
               PtPhObj_Get(Plan:AccountCode PartID:MemberID)
       Case B: AccountCode is product/account type
               Search member occurrences and match PtPhObj_De(700) = AccountCode
  -> Resolved Plan/Product/SubProduct/CrossPlan
  -> Other XML fields map through {CT}VT.TXNMACCUPD to PTPH data elements
  -> VtModel_Load_Event(EVENT:'MPERUPD', PLAN:ResolvedPlan, PART:MemberNumber)
  -> PTDE trailer pairs are populated with DE number/value
  -> ModelAdd writes transaction to VTRAN/folder
  -> Posting updates PTPH fields

Transaction Type Finding

The MemberAccountUpdate flow uses GetFolderDetails(... Transaction:'801') and VtModel_Load_Event(EVENT:'MPERUPD' ...). The update fields are loaded into PTDE trailer pairs beginning at transaction DE 055, with values at 060, then continuing by increments. 

GetFolderDetails(... Transaction: '801');
VtModel_Load_Event(EVENT: 'MPERUPD' PLAN: EmployerNumber PART: MemberNumber DATE: TradeDate);
VtTdTi_SetDe(DENUM:CardDe VALUE:CardDeNum TRLR:CardTab);
VtTdTi_SetDe(DENUM:CardDe VALUE:CardValue TRLR:CardTab);
The header comment in {FN}VT.TXNMACCUPD says "Output T813 Cards", but the confirmed code path uses the T801-style model/folder and PTDE trailer build. Do not rely on the comment alone.

What Is Persisted

Question Answer
Is AccountCode stored as a DE### by this flow? No confirmed direct storage. It is not present in {CT}VT.TXNMACCUPD as an update field.
Is it used as fund or source? No. In this flow it is account/occurrence selection context. No confirmed mapping to FUND, SOURCE, or PTPF.
What object identifies the account? PTPH, using the resolved plan/account occurrence plus MemberID. Product matching uses PTPH700.
What values are persisted? The other XML child fields mapped by {CT}VT.TXNMACCUPD, such as BenefitCategory -> PTPH643, InsuranceGroup -> PTPH627, IncomingStrategyElection -> PTPH636, etc.

UI Verification

To verify the outcome in Omni UI:

  • Open the member's account/occurrence identified by resolved PlanId + MemberID.
  • Confirm product/sub-product context on the member account, especially PTPH700 and PTPH701.
  • Check pending or posted transaction history for a T801 member account update with PTDE rows.
  • Verify the specific updated PTPH fields, such as PH636, PH643, PH627, etc., depending on the XML payload.

Bottom Line

AccountCode is best described as the external account selector. It may already be the Omni plan/account occurrence id, or it may be a product code that must be translated to the member's real occurrence. The actual account is the resolved PTPH occurrence, identified by PlanId + MemberID.
